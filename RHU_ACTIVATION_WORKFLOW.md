# RHU Account Activation Workflow

## Overview

This document describes the complete RHU (Rural Health Unit) account activation workflow. When a system administrator approves an RHU application, the system automatically sends an account setup email with a token-based link. The RHU then sets their password through a secure form before gaining access to the system.

## Architecture

### Components

1. **SystemAdminController** (`app/Http/Controllers/Admin/SystemAdminController.php`)
    - Handles RHU approval from admin dashboard
    - Generates username: `RHU_` + 8-character UUID prefix
    - Creates Firebase Auth user (password-less initially)
    - Calls `RhuAccountSetupController::sendSetupEmail()` to send invitation

2. **RhuAccountSetupController** (`app/Http/Controllers/Auth/RhuAccountSetupController.php`)
    - Manages token generation and validation
    - Displays password setup form
    - Processes password submission
    - Activates RHU account in Firebase and Firestore

3. **RhuAccountSetupEmail** (`app/Mail/RhuAccountSetupEmail.php`)
    - Mailable class for sending account setup emails
    - Professional HTML template with styling

4. **Email Template** (`resources/views/emails/rhu-account-setup.blade.php`)
    - HTML email with setup instructions
    - Includes setup link and 24-hour expiration warning

5. **Setup Form** (`resources/views/auth/rhu-setup.blade.php`)
    - Public password setup form (token-protected)
    - Password validation and confirmation

6. **Database Table** (`rhu_setup_tokens`)
    - Stores one-time setup tokens
    - Tracks expiration and usage
    - 24-hour token validity

## Database Schema

### rhu_setup_tokens Table

```sql
CREATE TABLE rhu_setup_tokens (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    rhu_id VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL,
    token VARCHAR(255) UNIQUE NOT NULL,
    expires_at TIMESTAMP,
    used_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX rhu_id_token (rhu_id, token),
    INDEX email (email)
);
```

## Workflow Steps

### Step 1: Admin Reviews Pending RHU Application

1. Admin navigates to System Admin Dashboard (`/admin/system-admin/dashboard`)
2. Views list of pending RHU applications
3. Reviews RHU details (name, email, contact info, etc.)
4. Clicks "Approve" button on desired RHU

### Step 2: Approval Triggers Email Invitation

When admin clicks approve:

```
SystemAdminController::approveAndSendCredentials()
  ↓
1. Generate username: RHU_XXXXXXXX
2. Create Firebase Auth user (no password yet)
3. Update Firestore:
   - username: RHU_XXXXXXXX
   - uid: <firebase_uid>
   - status: pending_setup
   - approved_by: <admin_id>
   - approved_at: <timestamp>
4. Call RhuAccountSetupController::sendSetupEmail()
   ↓
5. Generate unique 60-character token
6. Store token in rhu_setup_tokens table
   - expires_at: 24 hours from now
7. Send email via Mailtrap
8. Log approval and email sending
```

### Step 3: RHU Receives Email

The RHU receives an email containing:

- Welcome message with their name
- Generated username (e.g., `RHU_A1B2C3D4`)
- Clear call-to-action: "Set Your Password"
- Setup link: `https://yourdomain.com/setup-account/TOKEN`
- Warning: "This link expires in 24 hours"

### Step 4: RHU Clicks Setup Link

When RHU clicks the setup link:

```
GET /setup-account/{token}
  ↓
RhuAccountSetupController::showSetupForm()
  ↓
1. Look up token in rhu_setup_tokens table
2. Validate:
   - Token exists
   - Token hasn't been used (used_at is NULL)
   - Token hasn't expired (expires_at > now)
3. Display password setup form
   - Pre-filled email (read-only)
   - Password input field
   - Password confirmation field
4. Include hidden token field for submission
```

### Step 5: RHU Sets Password

When RHU submits the form:

```
POST /setup-account
  ↓
RhuAccountSetupController::handleSetup()
  ↓
1. Validate token (same checks as showSetupForm)
2. Validate password:
   - Minimum 8 characters
   - Confirmation matches
   - Required field
3. Update Firebase Auth:
   - Set password for user account
4. Update Firestore RHU document:
   - status: active
   - password_setup_at: <timestamp>
5. Mark token as used (set used_at timestamp)
6. Redirect to login page
7. Log account activation
```

### Step 6: RHU Logs In

RHU can now log in with:

- **Username:** `RHU_XXXXXXXX` (generated by admin)
- **Password:** Set during account setup
- **Email:** Confirmed during approval

## Routes

### Public Routes (No Authentication Required)

```php
GET  /setup-account/{token}   -> RhuAccountSetupController@showSetupForm   (rhu.setup-password)
POST /setup-account           -> RhuAccountSetupController@handleSetup     (rhu.setup-password.store)
```

```php
POST /admin/system-admin/approve-rhu/{id}  -> SystemAdminController@approveAndSendCredentials
```

## Error Handling

### Invalid or Expired Tokens

If a token is:

- **Not found:** "Invalid or expired setup link."
- **Already used:** "This setup link has already been used."
- **Expired:** "This setup link has expired. Please contact the administrator."

**Recovery:** Admin must send a new invitation email (creates new token).

### Password Validation Errors

- Password too short: "Password must be at least 8 characters."
- Confirmation mismatch: "Password confirmation does not match."
- Firebase error: "Account setup failed. Please contact the administrator."

### Automated Testing (Unit Test Example)

```php
// tests/Feature/RhuActivationTest.php

public function test_admin_can_send_setup_email()
{
    // Create pending RHU
    $rhu = $this->createPendingRhu();

    // Admin sends setup email
    $response = $this->post('/admin/system-admin/approve-rhu/' . $rhu['id']);

    // Verify response
    $response->assertStatus(200);
    $response->assertJsonPath('success', true);

    // Verify token created in database
    $this->assertDatabaseHas('rhu_setup_tokens', [
        'rhu_id' => $rhu['id'],
        'email' => $rhu['email'],
    ]);

    // Verify email would be sent (using Mail::fake() in actual test)
}

public function test_rhu_can_setup_account_with_valid_token()
{
    // Setup: Create token
    $token = $this->createSetupToken();

    // RHU submits form
    $response = $this->post('/setup-account', [
        'token' => $token,
        'password' => 'ValidPassword123!',
        'password_confirmation' => 'ValidPassword123!',
    ]);

    // Verify redirect to login
    $response->assertRedirect('/login');

    // Verify token marked as used
    $this->assertNotNull(
        DB::table('rhu_setup_tokens')
            ->where('token', $token)
            ->first()
            ->used_at
    );
}

public function test_expired_token_is_rejected()
{
    // Create expired token
    $token = $this->createExpiredSetupToken();

    // Try to access setup form
    $response = $this->get('/setup-account/' . $token);

    // Should redirect with error
    $response->assertRedirect('/login');
    $response->assertSessionHasErrors('error');
}
```

## Troubleshooting

### Email Not Being Sent

1. **Check Mailtrap Configuration**
    - Verify API key in `.env`
    - Verify inbox ID
    - Check Mailtrap logs

2. **Check Laravel Logs**
    - Location: `storage/logs/laravel.log`
    - Search for "Setup email" entries

3. **Check Firebase**
    - Verify user was created in Firebase Console
    - Check user authentication settings

### Setup Link Not Working

1. **Token Not Found**
    - Verify token in database: `SELECT * FROM rhu_setup_tokens`
    - Check token hasn't been marked as used

2. **Token Expired**
    - Check `expires_at` timestamp
    - Ask admin to send new invitation

3. **Database Connection**
    - Verify `rhu_setup_tokens` table exists
    - Run: `php artisan migrate`

### Account Not Activated

1. **Check Firebase**
    - Verify password was set
    - Verify user UID in Firestore RHU document

2. **Check Firestore**
    - Verify RHU status is "active"
    - Verify username is set

3. **Check Logs**
    - Look for errors in Laravel logs
    - Check Firebase admin SDK errors

## Security Considerations

### Token Security

- ✅ 60-character random tokens (cryptographically secure)
- ✅ Single-use tokens (marked as used after first submission)
- ✅ Time-limited tokens (24-hour expiration)
- ✅ Unique constraint on tokens (no duplicates)
- ❌ Tokens sent via email (inherently less secure, but industry standard)

### Password Security

- ✅ Minimum 8 characters enforced
- ✅ Firebase handles password hashing
- ✅ Password never stored in plain text
- ✅ HTTPS required in production

### Additional Recommendations

1. **Monitor Token Generation**
    - Log all token creations
    - Alert on suspicious patterns

2. **Add Rate Limiting**
    - Limit setup attempts per IP
    - Prevent token enumeration attacks

3. **Audit Trail**
    - Log all approvals and activations
    - Track admin who approved
    - Track timestamp of activation

4. **HTTPS Enforcement**
    - Ensure production uses HTTPS
    - Include Secure flag on session cookies
